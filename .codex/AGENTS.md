# AGENTS.md

このファイルは、Codex が作業するときの共通ガイドです。

## 基本方針

- 変更前に、既存のコード・設定・ドキュメントを確認する。
- 大きな書き換えよりも、小さく焦点の合った変更を優先する。
- ユーザーの未コミット変更を尊重する。関係ない変更を戻したり、上書きしたり、整形したりしない。
- 既存のスタイル、命名、ディレクトリ構成に合わせる。
- レビューしやすく、必要なら戻しやすい変更にする。
- 依存関係、ツール、抽象化は、明確に必要な場合だけ追加する。

## 作業手順

1. 依頼内容を理解し、関係するファイルを確認する。
2. 編集が発生しそうな場合は、事前にリポジトリの状態を確認する。
3. 依頼を満たす最小限で安全な変更を決める。
4. タスクに必要なファイルだけを編集する。
5. 変更内容に合った検証を行う。
6. 変更内容と、実行した検証または実行できなかった検証を簡潔に報告する。

## リポジトリの安全性

- 未コミット変更は、自分がこのタスクで作ったものを除き、すべてユーザーの作業として扱う。
- 明示的に依頼されない限り、`git reset --hard`、`git checkout --`、広範囲の `rm` などの破壊的なコマンドは実行しない。
- タスク上必要な場合を除き、ファイルの移動やリネームは行わない。
- 明示的に依頼されない限り、コミット、プッシュ、Pull Request の作成は行わない。
- リポジトリ外のファイルを変更する可能性があるコマンドや、ネットワークアクセスが必要なコマンドは、必要性を説明してから実行する。

## 編集方針

- 可能な限り、プレーンテキストで管理しやすい形式を保つ。
- ファイルが既に別の文字セットを使っている場合や、内容上必要な場合を除き、ASCII を基本にする。
- コメントは、意図や挙動が読み取りにくい箇所を補足する目的でだけ追加する。
- 新しい仕組みを持ち込む前に、既存のスクリプトや設定パターンを優先する。
- タスクと無関係なフォーマット変更は避ける。
- 生成ファイル、ロックファイル、ツール出力は、タスクで必要な場合だけ変更する。

## シェルとスクリプト

- テキストやファイルの検索には、可能な限り `rg` を使う。
- シェルスクリプトでは、必要に応じて防御的な書き方を使う。
  - エラーで停止すべきスクリプトでは `set -euo pipefail` を使う。
  - パスやユーザー入力を含む値はクォートする。
  - スクリプト自身の場所を基準にした明確な相対パス処理を行う。
- セットアップやインストール用スクリプトは、繰り返し実行しても壊れにくい形にする。
- リポジトリ内で既に使われている、または前提として明記されている場合を除き、特定のコマンドが存在すると決めつけない。

## 検証方針

変更内容に対して、必要十分な範囲で検証する。

- ドキュメントだけの変更: 表記、リンク、Markdown の体裁を確認する。
- シェルスクリプトの変更: `zsh -n path/to/script.sh` などで構文チェックする。
- 設定ファイルの変更: 利用ツールにフォーマッタやバリデータがあれば使う。
- アプリケーションコードの変更: 既存のテスト、lint、型チェック、ビルドのうち最も近いものを実行する。

検証を省略した場合は、その理由を報告する。

## Dotfiles に関する注意

dotfiles は、どの環境でも同じ開発体験を再現するための個人設定リポジトリとして扱う。

- 依頼されていないマシン固有の設定は変更しない。
- セットアップスクリプトは再実行しやすい状態を保つ。
- シークレット、トークン、プライベートなパス、特定マシン専用の値をハードコードしない。
- `.zshrc.local` のようなローカル上書き用ファイルを尊重する。
- シンボリックリンクの作成や更新は、実行前後の挙動を確認しやすい形にする。

## コミュニケーション

- 何をなぜ変更したかを簡潔に説明する。
- 触ったファイルと実行した検証を明記する。
- 前提、リスク、残作業があれば隠さず伝える。
- 依頼が曖昧でも低リスクな標準方針がある場合は、それを採用し、前提を説明する。
